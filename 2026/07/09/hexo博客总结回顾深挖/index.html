项目整体回顾

第一层：项目概览与环境准备

问： 我们做的到底是个什么东西？它是怎么工作的？
答： 我们做的叫「静态博客」。它由一系列已经生成好的HTML、CSS、JS文件组成，当用户访问时，服务器直接把这些文件发给浏览器，浏览器解析渲染成页面。没有数据库，没有服务端动态计算，所以加载快、安全、部署简单。

问： 为什么需要Node.js和npm？它们分别干什么？
答： 这个项目里的所有工具（Hexo、插件等）都是用JavaScript写的，而JavaScript需要在Node.js这个运行环境里才能执行。npm是Node.js的包管理器，用来下载和管理这些工具和依赖库。

问： Git和GitHub在项目中扮演什么角色？两者有什么区别？
答： Git是版本控制软件，能记录你对代码的所有修改。GitHub是一个网站，用来托管Git仓库，充当代码的远程备份和协作平台。我们用Git将代码推送到GitHub，然后用它提供的GitHub Pages或Actions功能来部署。

问： YAML文件是干嘛的？为什么配置文件要用它？
答： YAML是一种数据序列化格式，用来写配置文件（比如_config.yml）。它的特点是缩进表示层级，可读性非常好，适合人类手写和阅读。程序读取这个文件就能获得配置参数。

问： Markdown是编程语言吗？它和HTML什么关系？
答： Markdown是轻量级标记语言，不是编程语言。它设计的目的是易读易写，让你专注于内容。Hexo会把它转换成包含正确样式的HTML文件，这个过程叫渲染。

第二层：本地开发与预览

问： hexo init、hexo generate、hexo server这几个命令具体做了什么？
答：

  • hexo init：初始化一个博客骨架，帮你建好所有目录和必要的配置文件。
  • hexo generate：构建命令，它读取你的所有Markdown文章和配置，生成整个网站的静态HTML文件，存放到public文件夹里。
  • hexo server：启动一个本地开发服务器，让你在http://localhost:4000预览网站，方便调试。

第三层：虚拟化与Linux基础

问： 虚拟机是什么？我们为什么需要它，而不是直接在Windows上装Linux？
答： 虚拟机是用软件模拟的一台完整电脑，可以安装独立的操作系统。我们需要它来搭建一个纯净的Linux学习环境，模拟真实的云服务器，并且方便通过克隆创建多台机器，而不影响你自己的主机系统。

问： 虚拟机的NAT和Host-Only网络模式是什么意思？为什么需要双网卡？
答：

  • NAT模式：虚拟机通过宿主机上网，相当于躲在宿主机后面，外部网络无法直接访问它。我们用来让虚拟机安装软件。
  • Host-Only模式：宿主机和虚拟机之间建立一个独立的内部网络，和外部隔离。我们用来实现虚拟机之间互相通信。
  • 双网卡：就是为了让虚拟机既能安装软件（走NAT），又能相互访问（走Host-Only）。

问： Linux里的sudo和root是什么？为什么很多命令前要加它？
答： root是Linux里的超级管理员，拥有最高权限。sudo是让普通用户临时以root身份执行后面的命令，用来安装软件、修改系统配置等。加它是为了获取执行所需的高权限。

第四层：远程管理与部署

问： SSH是什么？它解决了什么问题？
答： SSH是一种加密网络协议，用来在不安全的网络上安全地远程登录和执行命令。它解决了明文传输密码的风险，让你能安全地控制远端的Linux服务器（比如我们的node2）。

问： SSH密钥认证比密码认证好在哪里？我们生成的公钥和私钥分别是什么？
答： 密钥认证更安全（破解难度极大），也更方便（可免密登录）。它使用一对密钥：公钥（锁）放在你想登录的服务器（如node2）上，私钥（钥匙）由你保管（在node1上）。登录时，服务器用公钥验证你手里的私钥，配对成功就放行。

问： scp和rsync都是传文件，有什么区别？
答： scp是简单复制，每次都会把指定文件全部传一遍。rsync是增量同步，它会比较源和目标的差异，只传输有变化的部分，所以效率更高，尤其适合频繁同步整个文件夹。

第五层：Web服务器与部署上线

问： Nginx是一个什么软件？在我们项目中扮演什么角色？
答： Nginx是一个高性能的Web服务器。在我们项目中，它扮演的角色是 “文件服务员” ，监听80端口，接收浏览器请求，然后从/var/www/blog目录里找到对应的HTML文件返回给浏览器，让我们的博客能被访问到。

问： GitHub Actions是什么？它怎么让我们实现”自动化”的？
答： GitHub Actions是GitHub提供的CI/CD（持续集成/部署）平台。它让你可以定义一套自动化流程（比如：当git push发生时，自动检出代码、安装依赖、构建网站、推送到Pages仓库）。我们把部署步骤写成.yml文件提交，之后的每次推送都由它自动执行。

问： 这套”自动部署”流程的具体步骤是什么？
答： 自动部署流程核心是.github/workflows/deploy.yml文件定义的：

  1. 触发：你git push到my-blog01仓库。
  2. 运行环境：GitHub启动一台临时Ubuntu服务器。
  3. 检出代码：把你的源码下载到临时服务器。
  4. 安装依赖：执行npm install安装Hexo等依赖。
  5. 构建网站：执行hexo generate生成public文件夹。
  6. 部署：将public文件夹内容推送到tristan1116.github.io仓库。

问： 什么是CI/CD？这个概念在我们项目里怎么体现的？
答： CI（持续集成）和CD（持续部署）是自动化开发和上线的理念。在项目里，你每次推送代码都是一个 “集成” 动作，之后GitHub Actions自动执行构建、测试、部署这一整套流程，就是持续集成与持续部署的实现。

项目总览与概念全景图

问： 现在你能用一句话总结，这个博客项目到底是怎么运作的吗？

问题深究

问题一：HTML、CSS、JS文件；前端后端

一、HTML、CSS、JS 到底是什么？

它们不是数据，而是构成网页的三种核心代码，各司其职。

文件类型 通俗理解 它在做的事 你项目中的例子
HTML 网页的骨架和内容 负责“有什么内容”。比如标题、段落、图片、链接。 Hexo 生成的 .html 文件
CSS 网页的衣服和样式 负责“长什么样”。比如字体、颜色、间距、布局。 主题里的 .css 文件（如 main.css）
JS 网页的大脑和动作 负责“能做什么事”。比如点击按钮弹出菜单、搜索功能。 主题里的 .js 文件（如 search.js）

在你的项目里，Hexo 把 Markdown 内容填充进主题模板，最后生成的就是这三种文件的组合，交付给浏览器。浏览器拿到它们，解析、渲染，就变成了你看到的漂亮页面。

二、它们是前端项目需要的数据吗？

不完全是。 更准确地说，它们是构建用户界面的代码。

我们对比一下就容易理解了：

概念 作用 例子
HTML/CSS/JS 界面代码。定义“如何展示”和“如何交互”。 一个按钮的样式、点击后的搜索逻辑。
数据 内容本身。定义“展示什么”。 一篇博客的标题、正文、作者、发布时间。

数据通常以 JSON、XML 或数据库记录的形式存在。在你的项目中，Markdown 文件就是数据，它记录了文章内容；而 Hexo 这个工具，就是负责把 Markdown 数据和 HTML/CSS/JS 模板结合起来，生成最终的网页文件。

三、与之相对的有哪些？

与之相对的，通常指后端技术。我们可以从几个维度来理解“相对”：

对比维度 前端（你问的这部分） 后端（与之相对）
运行位置 浏览器（用户端） 服务器（云端）
主要语言 HTML, CSS, JavaScript Python, Java, Go, PHP, Node.js
核心任务 负责展示和交互（用户看得见） 负责业务逻辑和数据（用户看不见）
典型功能 点击按钮、页面跳转、动画 用户登录验证、读取数据库、处理支付
你的项目 你访问的博客页面（全静态） 几乎没有。但如果你用云服务器部署，后端就是 Nginx

基于这个分类，一个完整的网站应用通常会分为：

  1. 前端（客户端）：你写的 HTML/CSS/JS 代码。
  2. 后端（服务器端）：提供数据的 API 接口或生成页面的程序。
  3. 数据库：真正存储数据的地方（如用户信息、文章内容）。

四、举一反三：知识拓展

从问题出发，可以衍生出两个重要的概念：

1. 静态网页 vs 动态网页

  • 静态网页：服务器上存着现成的 .html 文件，用户请求什么就发什么。你现在的博客就是完全静态的。
  • 动态网页：服务器上没有现成的 .html，用户请求时，后端程序（如 PHP、Python）实时从数据库读取数据，临时拼装出一个 .html 返回。

2. 数据格式：JSON
如果后端要向前端提供数据，它们最常用的语言就是 JSON（JavaScript Object Notation）。它轻量、易读，是目前 Web API 的事实标准。以后你学后端，打交道最多的就是它。

总结一下：HTML/CSS/JS 是构成页面的界面代码，不是数据。在开发领域，它们属于前端，与之相对的通常是处理逻辑和数据的后端。你目前的项目是一个纯前端 + 静态生成的博客。

JavaScript 是一种轻量级的、解释型的、基于对象和事件驱动的脚本语言。

三、运行环境是什么？为什么需要运行环境？

运行环境 是指执行代码所需的所有基础设施，包括解释器、内存管理、API 接口等。

JavaScript 有两种主流运行环境：

运行环境 谁提供的 主要用途

为什么需要运行环境？

因为 JavaScript 本身只是一个语法规范，它不能自己做事。它需要依赖运行环境提供的 API 才能发挥功能：

在你的项目里：

问题三：Hexo,Node.js;现代Web开发；

一、从最底层开始：JavaScript 和运行环境

先明确一个根本事实：JavaScript 本身只是一套语法规则，它不能自己运行。它需要一个“容器”来执行它，这个容器就是运行环境。

运行环境 作用 类比
浏览器 执行前端 JS，操作 DOM，响应用户事件 像一个舞台，让 JS 表演“交互”
Node.js 执行后端 JS，读写文件，启动服务器 像一个工坊，让 JS 做“工具”能做的工作

你的项目里，Hexo 就是在 Node.js 这个工坊里运行的。

关系链： JavaScript 语言 → 需要运行环境（Node.js/浏览器）→ 才能执行具体任务

二、Hexo 的本质是什么？

Hexo 的本质是一个“静态站点生成器”（Static Site Generator，SSG）。

它的工作流程是：

  1. 输入：读取你的 Markdown 文章 + 主题模板 + 配置文件
  2. 处理：在 Node.js 环境中，把 Markdown 转换成 HTML，把数据填充进模板
  3. 输出：生成一套完整的 HTML/CSS/JS 文件（也就是你的博客）

所以，Hexo 是一个运行在 Node.js 上的“构建工具”，它把“内容+模板”编译成“静态网页”。

现在我们把所有概念放进一个完整的 Web 开发流程里，你可以看到每一步用到什么、为什么需要它。

类比理解： C/C++ 多线程像多个人同时干活（线程切换开销大，要协调好资源）；JavaScript 的事件循环像一个超级高效的秘书，你给他一堆任务（I/O 请求），他排队处理，干完一个就通知你。
